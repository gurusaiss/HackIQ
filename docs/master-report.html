1. Executive Project Overview

FieldDetails
Project NameHackIQ (originally CompeteIQ)
Problem SolvedDevelopers/students waste hours manually scouring Devpost, Kaggle, HackerEarth for competitions — HackIQ aggregates and personalises them via AI
Elevator Pitch"HackIQ is an AI-powered platform that discovers hackathons and competitions tailored to your skills, scores how well each one matches your profile, and helps you track your applications — all in one place."
Target UsersCS students, developers, data scientists, designers looking for hackathons, ML competitions, and design challenges
Main GoalReduce discovery friction — go from "where do I find competitions?" to personalised, scored results in one click
Key InnovationAI match scoring engine (0–100%) + multi-tier API security (Vite proxy → Vercel serverless) + 30-min profile-keyed cache
Business ValueSaves 2–3 hours/week of manual research; increases participation in competitions; trackable pipeline (Bookmarked → Applied → Won)
Technical ComplexityMulti-provider AI (Groq + Anthropic), secure API key handling without a backend server, Firebase Firestore with localStorage fallback, dynamic match scoring algorithm
Unique Selling Points1. Profile-keyed AI match scores  2. No dedicated backend (cost-free)  3. Offline-capable via localStorage  4. Vercel serverless for production key security

30-second interview explanation

"HackIQ is a React + Firebase SPA that uses Groq's LLaMA-3.3 model to discover hackathons and competitions personalised to your skills. You fill in your profile — skills, interests, experience level — and the app searches competitions across Devpost, Kaggle, HackerEarth and others, scores each one against your profile using a weighted algorithm, and caches results for 30 minutes. API keys never reach the browser — they're injected by a Vite proxy in development and a Vercel serverless function in production."

Non-technical explanation (HR/Recruiter)

"Think of HackIQ as a personalised job board, but for competitions. You tell it your skills — Python, React, machine learning — and it uses AI to find hackathons and coding contests that match exactly your background. It scores every competition out of 100 so you always see the best matches first. Your saved competitions are tracked across devices using Firebase's free cloud database."

2-minute detailed explanation

Problem: Finding the right competition requires checking 5–6 platforms manually, with no personalisation.

Solution: HackIQ uses a React SPA with no dedicated backend. In development, a Vite proxy intercepts API calls and injects secret keys server-side. In production on Vercel, a Node.js serverless function (api/search.js) handles the same. The app calls Groq's API (llama-3.3-70b-versatile) with a structured prompt that includes the user's full profile and instructs the model to return JSON-formatted competition data.

Match Scoring: Each competition gets a score computed client-side: 40 points for skill overlap, 30 for interest alignment, 20 for experience level fit, 10 for location preference.

Persistence: Firebase Firestore stores profiles and saved competitions. If no Firebase keys are configured, everything falls back to localStorage — making the app fully offline-capable.

Performance: Results are cached in localStorage with a 30-minute TTL keyed by a base64 hash of the user's profile fields. If you revisit without changing your profile, results load instantly from cache.

Tech: React 18 + Vite 6 + Tailwind CSS 3 + Firebase Firestore + Groq AI + Vercel. GitHub: gurusaiss/HackIQ.

11. Interview Preparation — Top 25 Questions

Q1. How does your app avoid exposing API keys in the browser?
Answer: In development, Vite's proxy intercepts requests to /api/groq/* and /api/anthropic/*, injects the Authorization header server-side using keys loaded via loadEnv (non-VITE_ prefixed, never bundled). In production, a Vercel serverless function (api/search.js) reads keys from process.env and forwards the call. The browser never sees the keys.
Follow-up: Why not use VITE_ prefix? → Vite statically replaces VITE_ vars in the bundle at build time — they're embedded in the JS file sent to the browser.
Q2. Why Firebase over Supabase?
Answer: Supabase free tier pauses projects after 7 days of inactivity. For a portfolio project, that means the database goes offline until manually resumed. Firebase Firestore never pauses, offers 50K reads/day free, and has a browser SDK that works without a backend server.
Q3. Explain your match scoring algorithm.
Answer: computeMatchScore() is a weighted function: 40 points for skill overlap (intersection of user skills and competition required skills, normalized), 30 points for interest alignment, 20 points for experience level match, 10 points for location preference. Returns a 0–100 integer. Runs client-side on every competition the AI returns.
Q4. How does the caching work?
Answer: When a search completes, results are stored in localStorage under a key derived from a base64 hash of the user's profile fields (skills, interests, experience, preferences). On next visit, getCachedResults() checks if the cached result exists and is less than 30 minutes old. If yes, it's returned immediately without an API call. If the user updates their profile, the save function deletes the cache key.
Q5. Why no backend server?
Answer: A dedicated server adds cost, infra management, and deployment complexity. Vercel serverless functions handle the only server-side requirement (API key security) without any configuration. Firebase Firestore replaces the need for a REST API for data persistence. This keeps the total hosting cost at $0.
Q6. How does the Vite proxy work?
Answer: vite.config.js uses loadEnv(mode, cwd, '') to load ALL env vars including non-VITE_ ones. The proxy config maps /api/groq/* to https://api.groq.com — when the frontend calls /api/groq/openai/v1/chat/completions, Vite rewrites the URL and injects the Authorization header using the server-side GROQ_API_KEY. This only works in development.
Q7. What happens if there's no API key configured?
Answer: searchApi.js has a multi-tier fallback: (1) Vite dev proxy, (2) Vercel /api/search serverless, (3) Direct browser call with VITE_ prefixed key if set, (4) 12 hardcoded mock competitions with real platform URLs. The app is always usable.
Q8. How does Firebase integration work with localStorage fallback?
Answer: getDb() returns null if VITE_FIREBASE_PROJECT_ID is not set. Every function (getProfile, saveProfile, getSavedCompetitions, etc.) checks if db is truthy — if null, it executes the localStorage equivalent. The interface is identical from the caller's perspective.
Q9. How did you handle the CompetitionModal scroll lock?
Answer: In CompetitionModal, useEffect sets document.body.style.overflow = 'hidden' on mount and restores it to '' on cleanup. An Escape key listener is also added and removed via the same effect. This prevents the page from scrolling while the modal is open.
Q10. How would you add authentication?
Answer: Add Firebase Authentication (Google OAuth). Replace the hardcoded 'local-user' ID with auth.currentUser.uid. Update Firestore security rules to allow read/write only when request.auth.uid matches the document's user ID field. Add a login/logout button in Navbar.

Deliverable 2: Project Knowledge Base FAQ

Why React + Vite over Next.js?

HackIQ is a fully client-rendered app with no SEO requirements (no public competition pages to index). Next.js adds SSR complexity without benefit. Vite is faster in development and the proxy feature is built-in — perfect for the API security use case.

Why not just use OpenAI?

OpenAI doesn't support browser CORS by default. Groq does — meaning in a pinch you can call it directly from the browser with a VITE_ key for demos. Groq is also free. OpenAI would still require the proxy/serverless architecture.

Biggest challenge?

Making API key security work without a dedicated backend. Vite's VITE_ convention is a trap — it looks like an env var but gets embedded in the bundle. Needed to understand loadEnv(), the proxy mechanism, and Vercel serverless all at once.

Biggest mistake?

Initially choosing Supabase without checking the free tier pause policy. This required a full database layer rewrite mid-project. Lesson: always check infrastructure limitations before writing integration code.

Biggest learning?

That "no backend" doesn't mean "no server-side code". Vite proxy and Vercel serverless ARE server-side code — just managed differently. Understanding this distinction is key to building secure frontends without a full backend.

What would you improve?

1. Add Firebase Authentication — replace the hardcoded user ID with a real auth identity. 2. Add real-time competition data via a scheduled scraper instead of AI-generated data. 3. Rename supabase.js to db.js for clarity. 4. Add Sentry for error monitoring.

How would you scale to 1M users?

Migrate to Next.js for server components. Add Firebase Auth. Upgrade Firestore to Blaze plan with per-user security rules. Add Redis cache (Upstash) for competition search results to share across users. Add a backend service (Node.js + BullMQ) for queue-based AI searches. Deploy on Vercel Pro or Cloud Run for higher function limits.

Deliverable 3: STAR Stories

Story 1: API Key Security Bug

Situation: Building HackIQ — needed to call Groq and Anthropic APIs from a React frontend.
Task: Ensure API keys are never exposed in the browser bundle or network requests visible to users.
Action: Discovered Vite's VITE_ prefix embeds env vars into the JS bundle at build time. Designed a two-tier solution: (1) Vite dev proxy that intercepts /api/* calls and injects Authorization headers server-side using non-VITE_ keys. (2) Vercel serverless function (api/search.js) for production — reads keys from process.env and forwards calls.
Result: Zero API keys in browser bundle. Verified by inspecting the built JS files. Works in both dev and production without any config changes.

Story 2: Database Migration (Supabase → Firebase)

Situation: HackIQ was built with Supabase for database persistence. Midway through development, discovered Supabase's free tier pauses projects after 7 days of inactivity.
Task: Migrate to a database that never pauses, without breaking the 3 pages that imported from the supabase.js module.
Action: Researched alternatives and chose Firebase Firestore (never pauses, 50K reads/day free, browser SDK). Rewrote supabase.js from scratch with Firebase — keeping the exact same function signatures (getProfile, saveProfile, getSavedCompetitions, etc.). Kept the filename as supabase.js to avoid any import changes across Discover, Profile, and Saved pages.
Result: Full migration completed in one session. Zero changes to page files. App now runs on a database that's always available, with no performance degradation.

Story 3: Fake Competition Data Problem

Situation: Product audit identified that the AI was returning generic, low-quality, potentially fake competition data with made-up URLs.
Task: Make the AI return real, accurate competition data with valid application links.
Action: Rewrote the system prompt in searchApi.js to include: today's exact date (so deadlines are realistic), real platform URLs for Devpost/Kaggle/HackerEarth/Unstop/HackerRank, a strict 12-field JSON schema with examples, and explicit instruction to only reference platforms with real competitions. Added 12 hardcoded fallback competitions with verified real URLs.
Result: Competition data quality significantly improved. Deadlines are realistic. URLs point to real platforms. Fallback ensures usability even when AI quality degrades.

Story 4: Result Caching Optimization

Situation: Every visit to the Discover page triggered a 2–3 second AI API call, burning API quota and creating a poor UX for repeat visitors.
Task: Eliminate redundant API calls while keeping results personalised per user profile.
Action: Implemented a localStorage cache with a 30-minute TTL. Cache key is a base64 hash of the user's profile fields (skills, interests, experience, location). getCachedResults() checks age before serving. Cache is cleared when profile is saved. Discover page restores from cache on mount — no search needed.
Result: Repeat page visits went from 2–3s to <50ms. API calls reduced ~80% during a typical session. Profile changes automatically invalidate and refresh results.
